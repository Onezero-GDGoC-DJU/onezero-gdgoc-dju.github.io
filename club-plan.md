# 동아리 홈페이지 방향 정리

이 문서는 동아리원이 빠르게 현재 방향을 이해할 수 있도록 정리한 간단한 메모입니다.

## 왜 정적 HTML만으로 끝내지 않나

- 지금은 작은 사이트지만, 앞으로 섹션과 페이지가 늘어나면 반복되는 마크업 관리가 불편해질 수 있습니다.
- 동아리원들이 프론트엔드 기초와 컴포넌트 기반 개발 방식을 함께 배우려면 React 구조를 경험하는 편이 좋습니다.
- 다만 GitHub Pages 배포는 정적 결과물이 가장 단순하므로, 개발은 프레임워크로 하고 배포는 정적으로 하는 방향이 적절합니다.

## 추천 기술 방향

- 개발 도구는 `React + Vite`
- 이유는 구조가 단순하고, 입문자가 이해하기 쉬우며, 빌드 결과를 정적으로 배포하기 쉽기 때문입니다.
- `Next.js`도 가능하지만, 현재 단계에서는 개념과 설정이 더 무겁습니다.

## 추천 폴더 구조

```text
src/
  components/
  pages/
  data/
  assets/
  styles/
```

- `components`: 헤더, 푸터, 카드, 버튼 같은 재사용 UI
- `pages`: Home, About, Activities, Join 같은 화면 단위
- `data`: 동아리 소개, 활동 목록, FAQ 같은 정적 데이터
- `assets`: 이미지와 아이콘
- `styles`: 공통 스타일

## 작업 원칙

- 처음에는 정적인 소개 사이트부터 만든다.
- 상태관리나 백엔드는 나중에 붙인다.
- 한 사람당 한 섹션 또는 한 컴포넌트 단위로 작업한다.
- GitHub Issue, 브랜치, Pull Request 흐름을 실제로 연습한다.
- 코드 리뷰에서는 결과물뿐 아니라 HTML 구조, CSS 작성 방식, props 사용도 함께 본다.

## 우선 만들 페이지

1. 메인 소개 페이지
2. 동아리 소개 페이지
3. 활동 또는 프로젝트 페이지
4. 참여 안내 페이지
5. FAQ 페이지

## 현재 진행 방향

- 우선 메인 화면에서 동아리의 성격과 참여 이유를 분명하게 보여준다.
- 이후 React + Vite 기반으로 옮길 때도 지금 정한 섹션 구조를 그대로 가져갈 수 있게 설계한다.
