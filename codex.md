# Codex 작업 지침

이 문서는 `codex cli`를 활용해 이 저장소를 작업할 때 인공지능이 반드시 준수해야 하는 기준을 정리한 문서입니다.

## 기본 원칙

- 프로젝트의 목표는 동아리 홈페이지 완성과 학습 경험 축적임을 항상 우선한다.
- 이 저장소가 GitHub Pages로 배포되는 정적 웹사이트 프로젝트임을 항상 고려한다.
- 결과물만 빠르게 만드는 것보다, 사람이 이해하고 이어서 수정할 수 있는 코드를 작성한다.
- 기존 코드와 문서의 맥락을 먼저 확인한 뒤 작업한다.
- 확인하지 않은 사실을 단정하지 않는다.
- 불필요하게 큰 변경보다 작고 검증 가능한 변경을 우선한다.

## 작업 방식

- 작업 전에 현재 파일 구조와 관련 코드를 먼저 확인한다.
- 변경 사항이 GitHub Pages 환경에서 자연스럽게 동작할 수 있는지 고려한다.
- 사용자 요청이 모호하면 임의로 크게 확장하지 말고, 현재 목표에 맞는 최소 변경부터 수행한다.
- 수정 범위와 영향을 스스로 점검한 뒤 파일을 변경한다.
- 기존 스타일과 구조가 있다면 우선 맞춘다.
- 새 기능을 추가할 때는 사용 목적이 드러나는 이름을 사용한다.

## 코드 작성 원칙

- HTML, CSS, JavaScript는 초심자도 읽을 수 있게 단순하고 명확하게 작성한다.
- 과도한 추상화, 불필요한 라이브러리, 복잡한 패턴 도입을 피한다.
- 하드코딩이 필요하더라도 현재 프로젝트 단계에서 이해하기 쉬운 형태를 우선한다.
- 중복이 심해지기 전까지는 무리한 공용화보다 가독성을 우선한다.
- 접근성, 의미 있는 마크업, 기본적인 반응형 대응을 고려한다.

## Git / GitHub 관련 원칙

- 브랜치, 커밋, 이슈, Pull Request 흐름을 학습하는 프로젝트라는 점을 항상 고려한다.
- 한 번에 너무 많은 내용을 바꾸지 않는다.
- 변경 이유를 설명할 수 있는 단위로 작업한다.
- 협업 중일 수 있으므로 다른 사람이 만든 변경을 함부로 되돌리지 않는다.
- 파괴적인 명령이나 대량 삭제는 명확한 요청 없이 수행하지 않는다.

## 문서화 원칙

- README나 기타 문서를 수정할 때는 프로젝트 목적과 참여 방식이 드러나게 작성한다.
- 필요할 경우 GitHub Pages 기반 프로젝트라는 점이 문서에 드러나도록 유지한다.
- 설명은 짧고 명확하게 작성한다.
- 초보자 기준에서 이해 가능한 표현을 우선 사용한다.
- 실제로 하지 않은 작업이나 검증하지 않은 내용을 문서에 적지 않는다.

## 응답 원칙

- 무엇을 확인했고 무엇을 수정했는지 간결하게 설명한다.
- 실행하지 않은 테스트나 검증은 수행한 것처럼 말하지 않는다.
- 문제가 있거나 확신이 부족한 부분은 숨기지 말고 명시한다.
- 사용자가 바로 다음 행동을 결정할 수 있도록 실질적인 정보를 제공한다.

## 금지 사항

- 존재하지 않는 기능, 파일, 완료되지 않은 작업을 완료된 것처럼 표현하지 않는다.
- 요청되지 않은 대규모 리팩터링을 진행하지 않는다.
- 프로젝트 학습 단계에 비해 지나치게 복잡한 구조를 강요하지 않는다.
- 비밀 정보, 토큰, 개인 정보를 코드나 문서에 남기지 않는다.
