# Gemini 작업 지침

이 문서는 `gemini cli`를 활용해 이 저장소를 작업할 때 인공지능이 반드시 준수해야 하는 기준을 정리한 문서입니다.

## 기본 원칙

- 이 프로젝트는 비정기 스터디 기반의 학습형 프로젝트임을 항상 고려한다.
- 이 저장소는 GitHub Pages를 통해 배포되는 정적 웹사이트 프로젝트임을 고려한다.
- 동아리 홈페이지 완성, 프론트엔드 기초 학습, GitHub / Git 숙달이라는 목표를 해치지 않는 방향으로 작업한다.
- 빠른 생성보다 사람이 이해하고 이어받기 쉬운 결과물을 우선한다.
- 기존 산출물의 의도와 현재 작업 범위를 먼저 파악한다.
- 불확실한 내용은 추정이라고 분명히 밝히거나 보수적으로 처리한다.

## 작업 방식

- 작업 시작 전 관련 파일과 문서를 먼저 읽고 맥락을 파악한다.
- 제안하거나 수정하는 내용이 GitHub Pages 환경에 맞는지 함께 고려한다.
- 요청 범위를 벗어나는 기능 추가는 자제한다.
- 한 번의 출력으로 끝내기보다, 실제 반영 시 유지보수 가능한 형태를 우선 제안한다.
- 변경 내용은 가능한 한 작고 명확한 단위로 나눈다.
- 사용자가 수정하거나 확장하기 쉬운 구조를 유지한다.

## 코드 작성 원칙

- 초심자가 읽고 학습할 수 있는 수준의 코드로 작성한다.
- 복잡한 패턴이나 과도한 자동 생성 코드를 그대로 밀어 넣지 않는다.
- 의미 있는 태그, 일관된 클래스명, 읽기 쉬운 구조를 우선한다.
- 프론트엔드 작업 시 기본적인 반응형 대응과 사용자 경험을 고려한다.
- 필요 이상으로 의존성을 늘리지 않는다.

## 협업 원칙

- Discord 중심으로 협업이 이루어진다는 점을 고려해, 공유하기 쉬운 변경 단위를 만든다.
- 이슈와 Pull Request로 논의될 수 있도록 변경 이유가 분명해야 한다.
- 다른 작업자의 변경 가능성을 고려해 충돌 위험이 큰 수정은 신중히 다룬다.
- 다른 사람이 이해하기 어려운 과도한 축약이나 생략을 피한다.

## 문서화 원칙

- 문서는 실제 작업 방식과 일치해야 한다.
- 필요하면 GitHub Pages 기반 배포 프로젝트라는 점을 문서에 명확히 반영한다.
- 참여 방법, 목표, 진행 방식 등 운영 정보는 오해 없이 명확하게 작성한다.
- 튜토리얼이나 설명 문구는 입문자 기준으로 작성한다.
- 확정되지 않은 정책은 확정된 것처럼 쓰지 않는다.

## 응답 원칙

- 변경 사항, 이유, 영향 범위를 간단명료하게 설명한다.
- 검증하지 않은 결과를 확정적으로 말하지 않는다.
- 스스로 한 일과 하지 않은 일을 구분해서 전달한다.
- 사용자가 다음에 무엇을 하면 되는지 자연스럽게 이어질 수 있게 설명한다.

## 금지 사항

- 프로젝트 목적과 무관한 화려한 구조나 과도한 기능을 임의로 추가하지 않는다.
- 설명 없이 대규모 파일 구조 변경을 하지 않는다.
- 사실 확인 없이 외부 정보나 규칙을 단정적으로 적지 않는다.
- 민감한 정보나 내부 접근 정보가 포함된 내용을 남기지 않는다.
